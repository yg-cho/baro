### Task 1: 루트 README.md + LICENSE + ARCHITECTURE.md

**Files:**

- Create: `LICENSE` (MIT, copyright 2026 yg-cho)

**README.md 필수 내용 (순서대로):**
1. 훅: `**baro** (바로) — Korean for "right away". Clone it, run it, it's on.`
2. 한 문단 정체성: fullstack TypeScript starter — Hono API + Next.js web, typed end to end with zero codegen, embedded PGlite so there is no database setup step
3. Quick start (검증된 두 경로):
   ```bash
   npx create-baro
   # or manually:
   git clone https://github.com/yg-cho/baro && cd baro
   pnpm install && pnpm dev
   ```
   web `http://localhost:3000` / api `http://localhost:8000`. 요구사항: Node >=22, pnpm 10
4. What you get: 표 — auth (Better Auth: email/password + optional Google/GitHub), admin dashboard (`/admin`, `pnpm seed:admin you@example.com password`), example feature (todos — "copy this to build yours"), typed RPC (Hono `hc`, zero codegen), one gate (`pnpm check` = typecheck+vitest+biome+playwright, local=CI)
5. Scripts 표: dev / check / seed:admin / test:e2e / smoke:create / db:generate(--filter @baro/db)
6. Stack 표 (스펙 §3 기준, 버전은 major만)
7. Project layout 트리 (apps/web features, apps/api modules, packages db/auth/shared, tooling/create-cli)
8. "Adding a feature" — ARCHITECTURE.md 링크 + 4파일 공식 티저
9. Recipes 링크 목록 (docs/recipes/)
10. License MIT

**ARCHITECTURE.md 필수 내용:**
1. 선언: "This file is for humans and for AI agents. Any tool that can read one file can learn this codebase's rules here." (스펙 §5 — AI 도구별 설정 파일 없음이 의도)
2. 기능 추가 공식 — todos를 정확한 파일 경로로 예시:
   ```
   packages/shared/src/schemas/<feature>.ts      # zod schema (single source)
   apps/api/src/modules/<feature>/routes.ts      # Hono routes (+ .test.ts)
   apps/web/src/features/<feature>/              # components + query hooks
   apps/web/src/app/<feature>/page.tsx           # route
   ```
   "Copy the todos feature. It is the formula."
3. 규칙 목록 (전부 리뷰에서 검증된 사실):
   - Router-level guards: `.use()` must be a separate statement on OpenAPIHono (chaining widens the type and drops `.openapi`) — see modules/admin, modules/todos
   - app.ts chain order is load-bearing for AppType: `.doc()` first, `.route()` last; `/openapi.json` intentionally has no CORS
   - Error envelope: `{ error: { code, message } }`; exception: better-auth's own endpoints return `{ code, message }` (plugin-owned, not reshaped)
   - Ownership pattern: UPDATE/DELETE with `WHERE id AND userId` in one query; foreign rows → 404 (never 403 — don't leak existence)
   - DB: schema in packages/db, `pnpm --filter @baro/db db:generate` after schema edits, migrations run automatically at api boot; PGlite default, `DATABASE_URL` switches to Postgres
   - env: apps/api loads `.env` via `src/env.ts` (shell env wins); web uses `NEXT_PUBLIC_API_URL`
   - Internal packages export TS source directly (no build step); `apps/web/tsconfig.json` deliberately does not extend the base (Next owns it)
   - requireAuth vs requireAdmin duplicated by design — unify at the 3rd consumer, not before
   - bigint counts: `Number()`-coerce aggregates (node-postgres returns strings)
   - seed-admin lowercases emails (better-auth normalizes)
4. Security notes: admin plugin's impersonation endpoint is enabled by default (role-gated, 1h session, no UI — disable via plugin options if unwanted); sessions are HttpOnly SameSite=Lax cookies — web and api must share a registrable domain in production (or use sameSite:"none"+Secure); CSRF via origin allow-list (`trustedOrigins`)
5. Testing: vitest per package (PGlite in-memory, 30s timeouts for CI), Playwright smoke on ports 3100/8100 (fresh DB per run), `pnpm check` is the single gate
6. openapi.json documents success responses only (by convention)

**검증 스텝:** README의 모든 명령 실제 실행 (`pnpm check`는 이미 그린이므로 생략 가능, quick start clone 경로는 로컬에서 `pnpm dev` 기동+curl 확인), 링크 파일 존재 확인. Commit: `docs: add readme, architecture guide and license`

### Task 2: docs/recipes/ 6종

**Files:**
- Create: `docs/recipes/postgres.md` — PGlite→Postgres 전환: `DATABASE_URL` 설정이 전부(코드 변경 0), docker 예시 한 줄, 마이그레이션 자동 적용 언급, seed:admin 재실행
- Create: `docs/recipes/deploy.md` — web은 Vercel(`apps/web`, `NEXT_PUBLIC_API_URL` env), api는 Railway/Fly(Node 22, `pnpm --filter @baro/api dev`가 아닌 프로덕션 기동 — `tsx src/index.ts`, env: DATABASE_URL/BETTER_AUTH_SECRET/BETTER_AUTH_URL/WEB_ORIGIN). **same-site 쿠키 주의 박스**: 다른 도메인이면 로그인 쿠키 안 감 — 같은 등록 도메인(서브도메인) 쓰거나 sameSite none+Secure
- Create: `docs/recipes/social-login.md` — Google/GitHub OAuth 앱 생성 경로, 콜백 URL `${BETTER_AUTH_URL}/api/auth/callback/<provider>`, env 키 넣으면 버튼 자동 등장(코드 변경 0)
- Create: `docs/recipes/orval.md` — `/openapi.json`이 이미 서빙 중, orval 설정 예시로 react-query 클라이언트 생성, "Hono RPC가 기본 — 외부 소비자/타 언어 클라이언트 필요할 때만"
- Create: `docs/recipes/email.md` — 개발은 콘솔 출력, 프로덕션 SMTP: packages/auth의 sendResetPassword/sendVerificationEmail 콜백 교체 지점 명시 (nodemailer/resend 예시 코드 골격)
- Create: `docs/recipes/scaling-types.md` — Hono RPC 타입 추론이 느려질 때: 증상(수백 라우트에서 tsc 느려짐), 대응(모듈별 AppType 분리, hc 클라이언트 프리컴파일 패턴 링크)

각 레시피: 목적 1문단 + 단계 + 검증 방법. FSD 전환 레시피는 **제외** (스펙에 있었지만 유지비 대비 수요 불명 — README에서 링크 안 하면 죽은 문서. 스펙 편차로 기록).

**검증 스텝:** postgres.md의 docker 명령 + DATABASE_URL 플로우 실제 1회 실행 (docker 없으면 문법 검증만 후 보고). Commit: `docs: add recipes for postgres, deploy, social, orval, email, type scaling`
